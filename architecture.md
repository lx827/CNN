## 2. 各模块职责

### 2.1 边端 (Edge)
**作用：** 就像医院的"采血护士"，负责从病人（风机）身上采集数据，送给检验科（云端）。

**实际情况：** 边端通常是安装在风机现场的工业网关或嵌入式设备，连接加速度传感器。

**本项目：** 用 Python 脚本模拟或读取真实 `.npy` 数据文件。支持两种模式：

**模拟模式**（`USE_REAL_DATA=false`）：生成逼真的风机齿轮箱振动信号，包含 4 种工况：
- **正常**：平稳基频 + 轻微噪声
- **齿轮磨损**：啮合频率增强 + 边频带
- **轴承外圈故障**：周期性冲击脉冲
- **轴不对中**：2 倍转频分量显著增大

**真实数据模式**（`USE_REAL_DATA=true`）：从 `.npy` 文件读取真实轴承振动数据
- 数据分组：H（健康）、I（内圈故障）、O（外圈故障）
- 设备分配：WTG-001→H，WTG-002→I，WTG-003→O，WTG-004→I，WTG-005→O
- 采样率 8192 Hz，10 秒窗口，81920 点/通道

**默认参数（可在 `edge/.env` 中修改，优先通过前端 Settings 页面动态配置）：**
- 采样率：**8192 Hz**（真实数据模式，由数据文件决定）/ 25600 Hz（模拟模式）
- 采集时长：**10 秒**（可云端动态调整）
- 原始点数：**81920 点/通道**（真实数据）/ 256000 点（模拟）
- 普通上传间隔：**10 秒**（可云端动态调整，支持秒/分钟/小时，最大 999 小时）
- 任务轮询间隔：**2 秒**（可云端动态调整，范围 3~60 秒）
- **数据压缩：由云端控制**（`compression_enabled` + `downsample_ratio`）
  - 真实数据模式默认不压缩（ratio=1）
  - 模拟模式默认 8 倍压缩（81920 → 10240）
- 默认设备：**5 台风机**（WTG-001 健康，WTG-002/004 内圈故障，WTG-003/005 外圈故障）

**云端动态配置机制：**
- 边端启动时从云端 `GET /api/devices/edge/config?device_id=xxx` 拉取配置（含 compression_enabled、downsample_ratio）
- 运行中每 **30 秒** 刷新一次配置
- 前端 Settings 页面修改后，最长 30 秒内同步到边端
- 支持 **批量应用到所有设备**（`PUT /api/devices/batch-config`）
- 本地 `.env` 中的值作为 fallback 兜底

### 2.2 云端 (Cloud)
**作用：** 就像医院的"检验科 + 医生"，负责分析数据、下诊断、存档案。

#### API 层
把数据库里的数据包装成 HTTP 接口，供前端调用。使用 **FastAPI** 框架，自带自动生成的 API 文档（Swagger UI）。

#### 分析引擎（核心）
```
┌─────────────────┐     ┌─────────────────────┐     ┌─────────────────┐
│  读取 sensor_data │ ──→ │  nn_predictor.py    │ ──→ │  神经网络输出    │
│  （最近60秒）     │     │  （预留接口）        │     │  健康度/故障概率 │
└─────────────────┘     └─────────────────────┘     └─────────────────┘
          │                        │
          │              模型未启用或失败
          ▼                        ▼
     ┌─────────────────────────────────────────────────────────────┐
     │  analyzer.py（回退方案入口）                                  │
     │  - 优先调用 DiagnosisEngine（engine.py）进行综合诊断           │
     │  - 异常时自动回退到 rule_based.py（规则诊断）                 │
     │  - 特征提取统一由 features.py 处理                            │
     └─────────────────────────────────────────────────────────────┘
```

**为什么预留神经网络接口？**
- 简化规则算法只能做演示，真实工业诊断需要深度学习
- 预留接口后，你可以独立训练模型，不需要改动系统其他代码
- 模型加载失败时**自动回退到简化算法**，系统不会崩溃

### 2.3 用户界面 (Web)
**作用：** 就像医院的"电子病历大屏"，医生/护士通过它查看所有病人的状态。

**改造点：** 之前是纯前端假数据，现在通过 Vite 代理调用云端 API 获取真实数据。

- **Monitor 时域图**：x 轴为**时间（秒）**，直观显示波形
- **Monitor 频域图**：x 轴为**频率（Hz）**，后端真实 FFT 计算
- **Dashboard 设备总览**：以卡片形式展示**所有设备**的健康度、状态、运行时长，点击卡片查看该设备的仪表盘、故障分布、部件详情
- **DataView 数据查看**：表格展示所有设备及其批次，点击批次默认加载时域波形；FFT、STFT、包络谱、统计指标（峰度/裕度/峭度等）需**点击按钮后按需计算**，防止过度消耗算力，支持删除单条或批量特殊数据

### 2.4 数据库
**默认 SQLite：** 零配置，数据保存在 `cloud/turbine.db` 文件中。适合开发、演示、单机运行。

**可选 MySQL：** 如果需要多机部署或数据量很大，修改 `.env` 即可切换。

## 3. 数据流详解

### 3.1 日常监测流（边端 → 云端 → 数据库）
```
边端采集脚本每 10 秒执行一次：
  1. 生成/读取 4 个通道的振动加速度信号（各 81920 点，8192Hz × 10s）
  2. 【压缩】峰值保持降采样 8x → 10240 点
  3. msgpack 序列化 + zlib 压缩 → 约 30 KB
  4. base64 编码后放入 JSON
  5. HTTP POST 到云端 /api/ingest

云端收到后：
  1. 校验数据格式，识别压缩/原图模式
  2. 解压（如果是压缩模式）
  3. 每个通道存一条记录到 sensor_data 表
     - 普通数据：最多保留 16 个批次，循环覆盖（batch_index 1~16）
     - 特殊数据：手动触发采集，batch_index 从 101 起自增，永不覆盖
  4. 通过 WebSocket 推送更新通知给前端
```

### 3.2 分析诊断流（云端内部）
```
云端后台线程每 30 秒执行一次：
  1. 读取最近 60 秒的 sensor_data（4 个通道）
  2. 【优先】调用 nn_predictor.predict()
     - 如果返回结果，直接使用神经网络输出
     - 如果返回 None，执行 analyzer._rule_based_analyze()
  3. 写入 diagnosis 表（最新诊断结果）
  4. 更新 devices 表（健康度、状态）
  5. 计算通道级振动特征（RMS/峰值/峭度/峰值因子/脉冲因子等）
  6. 通道特征超阈值 → 生成通道级振动特征告警（含 batch_index 关联到原始数据）
  7. 诊断结果异常 → 生成设备级诊断告警（含健康度阈值和故障概率详情）
  8. WebSocket 广播最新诊断结果和告警
```

### 3.3 用户查看流（前端 → 云端 → 前端）
```
Dashboard：
  GET /api/dashboard/ → 所有设备列表 + 每台最新诊断 + 告警统计
  前端以卡片形式展示所有设备健康度，点击后显示该设备的仪表盘和部件详情

Monitor：
  GET /api/monitor/latest → 每个通道最新 2560 个时域点 + FFT 频谱
  前端点击"请求采集"按钮 → 云端创建任务 → 边端轮询执行 → 采集完成后刷新
  支持优先显示特殊采集数据（prefer_special=true）
  时域图 x 轴：时间（秒）；频域图 x 轴：频率（Hz），后端真实 FFT 计算

手动采集流程：
  1. 前端 POST /api/collect/request?device_id=WTG-001 → 创建 pending 任务
  2. 边端 GET /api/collect/tasks?device_id=WTG-001 → 获取任务并执行
  3. 边端采集 8192Hz/10s 数据 → POST /api/ingest/（is_special=1, task_id=xxx）
  4. 云端分配 batch_index（101 起自增，不覆盖）→ 保存 → 自动分析
  5. 前端轮询 GET /api/collect/tasks/{id}/status → 显示进度 → 完成后刷新数据

DataView（数据查看）：
  GET /api/data/devices → 所有设备及批次列表（表格展示用，含 diagnosis_status：normal/warning/fault）
  GET /api/data/{id}/batches → 最近 50 个批次（含普通和特殊数据，含 diagnosis_status）
  GET /api/data/{id}/{batch}/{ch} → 原始时域数据（自适应采样率，默认自动加载）
  GET /api/data/{id}/{batch}/{ch}/fft → 实时 FFT（按需点击计算）
  GET /api/data/{id}/{batch}/{ch}/stft → 实时 STFT（按需点击计算）
  GET /api/data/{id}/{batch}/{ch}/envelope → 实时包络谱（按需点击计算）
  GET /api/data/{id}/{batch}/{ch}/order → 实时阶次谱（阶次跟踪分析，按需点击计算：通过频谱峰值法估计转频，时域→角域重采样后做 FFT，阶次轴以转频倍数为基准，支持转频搜索范围和每转采样点数配置）
  GET /api/data/{id}/{batch}/{ch}/cepstrum → 实时倒谱分析（Cepstrum，按需点击计算：FFT→对数幅度谱→IFFT，检测频谱中的周期性结构并自动标注峰值，支持最大倒频率配置）
  GET /api/data/{id}/{batch}/{ch}/stats → 统计特征指标（按需点击计算：峰值/RMS/峭度/偏度/裕度/波形因子/脉冲因子/峰值因子等；加窗统计量输出时序图，窗口大小和滑动步长可调）
  DELETE /api/data/{id}/{batch} → 删除整条批次（含关联诊断结果）
  DELETE /api/data/{id}/special → 删除该设备所有特殊数据

Diagnosis：
  GET /api/diagnosis/ → 最新诊断结果（健康度、故障概率、IMF能量）

Alarm：
  GET /api/alarms/?page=1&size=10 → 分页告警列表（支持 level/resolved/device_id 过滤）
  POST /api/alarms/{id}/resolve → 处理告警
  DELETE /api/alarms/{id} → 删除单条告警

Settings（配置管理）：
  GET /api/devices/{id}/config → 获取边端配置（采样率、采集时长、通道数、通道名称、上传间隔等）
  PUT /api/devices/{id}/config → 更新边端配置（支持 channel_names 通道名称映射）
  GET /api/devices/{id}/alarm-thresholds → 获取通道级振动特征阈值（含 effective_thresholds：用户配置与系统默认值合并）
  PUT /api/devices/{id}/alarm-thresholds → 更新通道级振动特征阈值（支持按设备独立设置，warning < critical 校验）

实时推送（WebSocket）：
  前端连接 ws://localhost:8000/ws/monitor
  云端在收到新数据 / 产生新告警时主动推送
```

## 4. 技术选型理由

| 模块 | 技术 | 理由 |
|------|------|------|
| 前端 | Vue 3 + Vite | 你已写好，不改 |
| 后端 | Python + FastAPI | 语法简单、自动生成 API 文档、原生支持异步/WebSocket |
| 数据库 | **SQLite**（默认） | 零配置、无需安装、文件即数据库，对零基础最友好 |
| 数据库 | MySQL（可选） | 成熟稳定，生产环境标准选择 |
| 边端 | Python | 与云端同语言，减少学习成本 |
| 神经网络 | ONNX / PyTorch / TF（预留） | 不绑定框架，你用什么训练都可以接入 |

## 5. 数据压缩方案

### 为什么需要压缩？

| 指标 | 数值 |
|------|------|
| 原始数据 | 4 通道 × 81920 点 × 8 字节 ≈ **2.5 MB** |
| 上传频率 | 每 10 秒一批 |
| 月流量 | **~6.5 TB**（不压缩） |

压缩后：**~30 KB/批**，月流量降至 **~80 GB**。

### 压缩流程

```
原始波形 81920点
    ↓ ① 峰值保持降采样（每N点取最大绝对值，防漏冲击）
10240点（默认 8x）
    ↓ ② msgpack 序列化（比 JSON 小 30~50%）
二进制
    ↓ ③ zlib 压缩（gzip 同级算法）
压缩包
    ↓ ④ base64 编码
JSON字符串 ← HTTP 上传
```

### 配置方法

编辑 `edge/.env`：
```
# 多设备配置（逗号分隔）
DEVICE_IDS=WTG-001,WTG-002,WTG-003,WTG-004,WTG-005

SAMPLE_RATE=8192         # 采样率 Hz
DURATION=10             # 采集时长 秒
DOWNSAMPLE_RATIO=8      # 降采样倍数（81920 → 10240）
COMPRESSION_ENABLED=true
```

改完后重启边端即可生效。**云端自动兼容压缩/原图两种格式。**

## 6. 神经网络预留接口详解

### 6.1 文件位置

```
cloud/
├── app/services/
│   ├── analyzer.py       ← 分析主入口，优先调用 nn_predictor
│   └── nn_predictor.py   ← 【你主要修改的文件】模型加载 + 推理
├── models/
│   └── (你的模型文件)     ← .onnx / .pt / .pb
└── .env                  ← NN_ENABLED=true / NN_MODEL_PATH=...
```

### 6.2 接口约定

**输入：**
```python
channels_data = {
```

### 6.3 接入步骤

1. 训练模型 → 输入 `[256000]` 或 `[32000]` 时域信号，输出 `[6]` 概率向量
2. 导出模型 → 放到 `cloud/models/`
3. 修改 `cloud/.env` → `NN_ENABLED=true`
4. 修改 `nn_predictor.py` → 补充 `_load_model()` 和 `predict()` 中的推理代码
5. 重启后端 → 系统自动使用神经网络

## 8. 扩展建议

**已实现（可直接使用）：**
- ✅ **多设备接入**：支持多台风机同时监测（Dashboard 显示所有设备健康度）
- ✅ **真实数据读取**：支持从 `.npy` 文件读取真实振动数据
- ✅ **采样率兼容**：支持任意采样率（8192Hz / 25600Hz 等），FFT/STFT/包络谱自动适配
- ✅ **通道级振动特征告警**：支持按设备/通道自定义 RMS、峰值、峭度、峰值因子的预警/严重阈值
- ✅ **边端离线检测**：设备超过 5 分钟无数据自动标记为离线，Dashboard 显示离线状态
- ✅ **诊断结果关联批次**：Diagnosis/Alarm 页面支持一键跳转到 DataView 查看原始数据
- ✅ **WebSocket 断线重连**：Monitor 页面接入 WebSocket 实时推送，断线后自动指数退避重连
- ✅ **数据导出**：DataView 支持将时域数据导出为 CSV
- ✅ **批量删除**：DataView 支持批量选择普通批次后一键删除
- ✅ **告警详情增强**：告警描述包含当前值、阈值对比、超标百分比、故障概率等
- ✅ **设备独立阈值**：每个设备可独立配置告警阈值，支持 null（禁用）和有效值
- ✅ **动态配置**：边端配置支持采样率、采集时长、通道数、通道名称等参数

**已扩展（最新实现）：**
- ✅ **参数化故障诊断**：支持配置齿轮齿数、轴承几何参数，自动计算啮合频率/轴承故障特征频率
- ✅ **多特征融合诊断**：时域统计 + 频谱特征 + 包络特征 + 阶次特征 加权融合
- ✅ **云端控制压缩**：Settings 页面可动态控制边端压缩开关和压缩比
- ✅ **批量设备配置**：一键将所有配置应用到所有设备

**待扩展：**
1. **神经网络升级**：训练 1D-CNN / ResNet / Transformer 模型替换简化算法
2. **真实传感器接入**：把 `signal_generator.py` 换成串口/NI-DAQ/Modbus 读取
3. **时序数据库**：高频原始数据用 InfluxDB，业务数据保留在 MySQL
4. **边缘智能**：在边端部署轻量模型，网络中断时也能本地告警
5. **自适应阈值校准**：基于历史健康数据自动学习设备基线，动态调整预警和严重阈值
6. **历史趋势分析**：增加设备健康度、振动特征指标的历史趋势曲线
7. **报表生成**：支持按时间段生成诊断报告（PDF/Excel）
8. **告警规则引擎**：支持更复杂的告警规则（多条件组合、持续时间、趋势判断等）
